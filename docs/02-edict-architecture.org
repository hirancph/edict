#+title: 02 — Edict Architecture Explained (For Beginners)
#+author: hirancph

* What is Edict?

Edict is a modular configuration framework for GNU Guix. 

Normally, when you configure a Linux system or GNU Guix, you have a massive file with thousands of lines of code. It becomes impossible to read.

Edict solves this by turning your configuration into *Lego blocks*. We call these blocks *Features*.

* The Lego Blocks: Features

Imagine you are building a Lego house. You have a "Living Room" block, a "Kitchen" block, and a "Roof" block. 

When you want to add Bluetooth, Wi-Fi, and NetworkManager to your computer, you don't write 50 lines of code in your main file. Instead, you just drop the =(networking-feature)= block into your configuration.

All features live in the `modules/edict/features/` directory.

* The Rulebook: Requires and Provides

Some Lego blocks can only be placed on top of other specific blocks. For example, you can't place a Roof block unless you have Walls.

In Edict, features clearly state what they *Provide* and what they *Require*.

#+begin_src scheme
  (edict-feature
   #:name 'desktop           ;; My name is Desktop
   #:provides '(desktop)     ;; I provide a desktop environment
   #:requires '(base))       ;; I REQUIRE the "base" feature to be installed first
#+end_src

Because of this, *you never have to worry about what order you list your features in your configuration*. Edict reads the rulebook and automatically sorts the blocks so that the foundation goes first, and the roof goes last. This is called a "Topological Sort" under the hood.

* Extension Targets

When a feature is added to your system, how does it actually tell the operating system what to install?

It contributes to *Targets*. Think of targets as buckets.
- There is a =system-packages-target= bucket.
- There is a =Groups-target= bucket.

When you add the =desktop-feature=, behind the scenes, that feature walks up to the =system-packages-target= bucket and throws =brightnessctl= inside. It walks up to the =groups-target= bucket and throws the "video" and "audio" groups inside.

When Edict builds your actual system, it just pours out all the buckets into the Guix Operating System.

* Value Sharing

Sometimes, features need to talk to each other. 
For example, the `nvidia-feature` needs to tell the rest of the system, "Hey, I am installed!" so that the Desktop environment knows to configure things specially for NVIDIA.

Features can publish *Values*:
#+begin_src scheme
   #:values `((has-nvidia? . #t))
#+end_src
Other features can ask the engine: "Hey, did any feature publish `has-nvidia?`" and Edict will answer `#t` (True).

* Summary of the Flow

1. Define a list of features you want for your computer in `hosts/vessel.scm`.
2. Edict reads the list and auto-sorts them based on their Requirements.
3. Edict lets the features share `Values` with each other.
4. Edict asks every feature to drop its configurations into the `Target` buckets.
5. The Builder (`build.scm`) takes those buckets and builds your exact Guix system!
